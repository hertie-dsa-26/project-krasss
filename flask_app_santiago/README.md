Module Descriptions and How They Connect


splitter.py contains prepare_data() and the Splitter class. prepare_data() loads the dataframe, drops the non-target health variables and identifier columns, and returns X, y and years. Splitter takes those arrays and produces time series folds via time_series_splits() and the final train/test split via get_test_split(). This is the first thing called in the pipeline.


preprocessing.py contains detect_categorical_columns(), SimpleImputer, StandardScaler, OneHotEncoder and Preprocessor. These are independent classes connected through composition inside Preprocessor,  not through inheritance. Preprocessor is the only class called from outside this file. It applies encoding → imputation → scaling in that order. The fit/transform separation is strictly enforced — fit_transform() is only ever called on training data.


krr.py  contains gaussian_kernel() and KernelRidgeRegression. I took it from the file that Sanjeev  wrote. The kernel function computes pairwise similarity between data points. KRR uses that similarity matrix to solve (K + λI)a = y for the dual coefficients and predicts with K_test @ a. There is an important note on one change I did on the kernel implementation below.


assessment.py contains Assessment with two methods  mean_squared_error() and r2_score(). This class was created by Sanjeev as well and it  computes scores and does not manage any CV loop or state. It is used only for final test set evaluation.


cross_validator.py contains CrossValidator which owns the full CV loop. It imports Preprocessor directly and creates a fresh instance per fold to prevent data leakage between folds. It calls Assessment internally for scoring and returns a dictionary with mse and r2 lists. start_fold controls which folds count toward hyperparameter selection.


train.py orchestrates the full pipeline for each health variable. It calls splitter.py, cross_validator.py, preprocessing.py, krr.py and assessment.py in sequence. After fitting the final model it saves both the KRR and the preprocessor together as a .pkl file via save_model(). It also contains load_model() which is imported by app.py.


scenarios.py contains the SCENARIOS dictionary and generate_scenario(). It takes the 2023 row for a selected county as a baseline, creates 10 synthetic (hypothetical for now) future rows and applies the corresponding climate changes. It returns a raw unscaled feature matrix that must go through preprocessor.transform() before prediction. County names are not unique across states so it uses state_abbr to uniquely identify a county (e.g. CLINTON COUNTY exists in Michigan, Iowa and New York). The app was initially crashing because I hadn't considered the possibility of duplicate names. Including the state fixes it, but we could also consider using the FIPS code if necessary to properly identify counties.


app.py is a minimal Flask app with two routes. GET / renders the form. POST /predict loads the model, generates scenario data, preprocesses and predicts. The full prediction flow is:


load_model(target)              → krr, preprocessor  from train.py
generate_scenario(...)          → X_future, years     from scenarios.py
preprocessor.transform(X)      → X_scaled            from preprocessing.py
krr.predict(X_scaled)          → y_pred              from krr.py


Changes from Original structure


The original implementation guidance suggested we used inheritance, where classes like Preprocessing and Assessment extended the Model class. In the current implementation I moved to composition instead. Preprocessor for example does not extend SimpleImputer, StandardScaler or OneHotEncoder, it owns instances of all three and calls them in the right order. The same logic applies across the whole pipeline. No class extends another. 

The practical reason for this change is that inheritance created some dependencies that were confusing for me . The current structure I think lets each class do more or less one single thing which I find easier to follow.

The original structure had a class called Model that combined data splitting and cross validation in one place. I made the following changes:


1. Model → Splitter : The class was renamed to Splitter to accurately reflect what it does, it only splits data inot X and y. It has no knowledge of models, scoring or preprocessing. 


2. Cross validation moved to its own file  Alex had suggested this as a possibility as well in the guidance and it ultimately made more sense to me. cross_validator.py is now completely independent. The reason is that cross validation needs to import Preprocessor to create fresh instances per fold, which would have created a circular import if it lived inside assessment.py or splitter.py. Making it a standalone class also makes the separation of concerns cleaner:


Splitter          → knows about data
Preprocessor      → knows about feature transformation
KRR               → knows about the model
Assessment        → knows about scoring
CrossValidator    → knows about all of the above and coordinates them
train.py          → knows about everything and runs the full pipeline


Cross Validation Approach


As we discussed in our last sprint retrospective, I used walk-forward time series cross validation rather than standard k-fold. 2023 is separated as the test year and the folds start from 2014 progressively until 2022:
Fold 0: train=2014,                validate=2015
Fold 1: train=2014-2015,           validate=2016
Fold 2: train=2014-2015-2016,      validate=2017
Fold 3: train=2014-2015-2016-2017, validate=2018
...
Fold 7: train=2014-2021,           validate=2022
Final:  train=2014-2022,           test=2023


The start_fold fix: Early folds have very small training sets (255 rows) and the data density jumps dramatically at 2018 (from 255 to 896 rows per year). This makes folds 0-4 produce highly unstable and unreliable MSE scores, which would mislead hyperparameter selection. To address this we only use the last 3 stable folds (validating on 2020, 2021 and 2022) for hyperparameter selection. The final model is still trained on the full 2013-2022 data.
