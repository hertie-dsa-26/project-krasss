# Coding and Development Practices
Taken from: _**RUBRIC.md**_

The following practices were derived from the 4 areas of the project and comply with the _"Exceeds Expectations"_ criteria.

## 1. Architecture & Design (25%)

* **High modularity:** components should be understood, tested, or replaced independently.
* **Thoughtful data structure choices:** should contain evidence that tradeoffs were considered.
* **Deliberate architectural decisions:** should be reflected in the overall design.


## 2. Implementation Quality (30%)

* **Code** is **clean** and **idiomatic**.
* **Complexity and limitations** of the from-scratch algorithm are considered.
* **Performance and edge cases** are considered and addressed.


## 3. Development Process (25%)

* **Code reviews** are **substantive**: should contain real feedback and iteration.
* **Test coverage** is **thorough**: should include edge cases.
* **Commit history** tells a clear **story** of the project's development.
* **Automation** is used to enforce **quality**.


## 4. Final Product (20%)

* **App** is **polished** and pleasant to use.
* **Visualizations** are **well-chosen** and **informative**.
* **End-to-end flow** (from exploration to model results) feels **seamless** and well thought through.


## Areas to improve/implement

* **Testing:** unit tests (single function or method), integration tests (two or more units working together), end-to-end tests (the whole application).
* **Data structure choices:** specify data types, e.g.: ``def factorial(n: int) -> int: . . .``.
* **More pull requests and code reviews** with substantial engagement, comments, and iteration.
* **Algorithmic efficiency:** we should be aware and perhaps try to improve the efficiency of our ML algorithm, e.g., by timing it.
* **Modularity:** ideally, our repository structure should be reconsidered, at least to some extent. We need to emphasize _importable modules_ rather than notebooks. Look below for a suggested structure. The tree itself was generated by ChatGPT after I explained the problem and my vision---we should definitely refine it together as it's merely a proprosal.
* **Repository--project alignment:** Note that in the suggested structure there's a single ``requirements.txt``. Right now, our repository looks like multiple mini-projects living inside one project, instead of one clean application. For example, we have a ``.venv`` inside ``app/`` and a separate ``uv.lock`` and a ``pyproject.toml`` in ``data_cleaning/``. Ideally, we would want: one repository = one Python project = one environment = one dependency file.
